Both protocol decoders parse the body BEFORE consulting frame.status, so a non-2xx response with a non-JSON body (proxy HTML, empty 5xx) is mis-classified.

- src/protocol/anthropic/decode.rs:15-19 — serde_json::from_slice(...).map_err(Transport)? runs before the `if let Some(status)` at line 24.
- src/protocol/openai/decode.rs:19-22 — error_value(&parse(&frame.data)?, status); the parse `?` fires before the status is used (parse maps any JSON failure to ErrorKind::Transport, decode.rs:199-204).

Effect: a 5xx with a non-JSON body exits 69 (Transport) instead of 70 (Provider{>=500}); a non-2xx generally loses the authoritative status. Contradicts spec §4.3 ('every other code -> Provider{status}') and the AGENTS.md north star 'Carry the fact; never reconstruct it from a lossy proxy' — frame.status IS carried, then dropped on the parse-error path.

Fix: when frame.status.is_some(), build the error from from_http_status(status) regardless of body parseability; the body parse becomes best-effort for message/provider_detail (fall back to empty message + None detail). Apply to BOTH decoders. Add a fixture/test: non-2xx + non-JSON body -> Provider{status} + correct exit.
