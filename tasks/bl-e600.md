Confirmed (auth-sso + error-exit review lanes). src/auth/refresh.rs:~59-62 collapses EVERY non-token-parseable refresh response to ErrorKind::Auth -> exit 77 (non-retryable, 'revoked or expired, re-login'), because the refresh path never peeks the HTTP status. So a transient token-endpoint 503/429 tells the operator their credential is dead and to re-run bz --login — wrong and alarming for a retryable transient.

Two parts:
1. CLEAR (do now): soften the user-facing message so a refresh failure does not assert 'revoked/expired' when it may be transient (e.g. 'token refresh failed; re-run bz --login if this persists').
2. OWNER SUB-DECISION (exit code): either (a) amend auth.md §6.2 to BLESS the single-path RefreshFailed->77 (mirrors the device-poll path; AGENTS.md low-mechanism — the operator action is the same regardless of status, so plumbing status through the refresh seam just to split codes is mechanism for a non-problem), or (b) plumb the status through and map 5xx->70 / 429->69. RECOMMEND (a): bless 77, fix the message, amend §6.2. Narrow blast radius (operator OAuth only).
