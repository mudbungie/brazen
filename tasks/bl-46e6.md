## Context

One-way-door review (2026-06-27). `brazen` ships as one crate: the `bz` binary is the
real product (users wire to the literal executable, unix-pipe philosophy); the `brazen`
lib is published so the pure pipeline can be embedded (architecture.md §1, §10). crates.io
`brazen` is still the empty 0.0.0 stub — nothing is published yet, so we have total
latitude. This is the window to nail it.

Problem: `src/lib.rs` re-exports nearly the whole internals (`parse`, `pump`, every
`Sink`, `query_from_request_line`, the OAuth wire builders, `jwt`, `parse_callback`, …),
almost certainly because the black-box `tests/` suite can only see `pub` items — i.e.
**test layout is driving the semver surface**. Plus zero `#[non_exhaustive]` anywhere in
`src/`. Every `pub` item becomes a 1.0 promise the moment we cut a real release.

## Decision (owner)

The binary is the real deliverable; users wire to the executable. The lib must still be
ship-shape before we ship. Narrow the public lib surface so the ONLY publicly exposed
interfaces are the ones actually reachable through the CLI — and make this a
**bidirectional invariant**: nothing doable through the CLI may be impossible via the lib,
AND nothing in the lib may be unreachable via the CLI. Exclusive parity, both directions.

## Deliverable

1. Record the invariant in architecture.md (living doc): "the public lib surface is
   exactly the capability set the `bz` CLI exposes — bidirectional exclusive parity."
2. Narrow `src/lib.rs`: demote everything not part of that surface to `pub(crate)`;
   relocate any `tests/` that depend on now-private items into in-crate `#[cfg(test)]`
   modules, so test layout stops driving the semver surface.
3. A **test model that asserts the parity and is forward-compatible to arbitrary new
   interface additions** — when a new capability is added to either side, the test
   enforces it appears on both without being rewritten per-capability. Derive both sets
   mechanically and assert set-equality; do NOT hand-maintain an allowlist.

## Open design questions (resolve in the task)

- Canonical enumeration of "CLI capability": flags + control ops + output modes + input
  channels + the canonical request/event fields they exercise.
- Canonical enumeration of "lib public surface": `cargo public-api` output, or a curated
  `pub use` manifest the test diffs against.
- The forward-compatible assertion is the hard part — prefer mechanical derivation of both
  sets over a hand-maintained list, so new interfaces can't slip in on one side only.

Type: **DESIGN** (living spec section + the parity test harness; the `pub`→`pub(crate)`
narrowing is the implementation the test then enforces).
