Feature gap surfaced reviewing the shipped CLI.

WHAT EXISTS: the thinking flag is Option<bool> (config/partial.rs:114) wired ONLY to the sink — 'emit reasoning before the answer under the text projection ... inert outside it.' Grep of every protocol encoder (anthropic/openai/openai_responses/google) confirms thinking is NEVER added to the OUTBOUND request body; it only gates whether thinking_delta events display in text mode. --json shows them regardless.

THE GAP: there is no budget_tokens / reasoning_effort field anywhere in the tree. Providers that require thinking be ENABLED with a budget (Anthropic extended thinking) or an effort level (OpenAI reasoning) cannot be driven from brazen's flags/config. Today the only escape is hand-authoring the request-body knob via the provider row's body_defaults or the piped canonical-request JSON.

DESIGN QUESTION (not a foregone yes): does brazen WANT a request-time reasoning knob, or is body_defaults/stdin the intended single home? AGENTS.md cautions: new flags are a smell, prefer an existing explicit signal; single source of truth; build less. If added, weigh a scalar (budget tokens) vs an effort enum vs leaving it to body_defaults. Whatever we decide, the arch doc (§5.3) and --help must state the boundary so 'thinking' is not read as a request knob it is not.

Deliverable per AGENTS.md: a decision recorded in the arch doc (living document); code only if the decision is to add the knob.
