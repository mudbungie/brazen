HttpTransport::new (src/bin/main.rs) hardcodes connect=30s and recv_response=120s and leaves the response body unbounded. Two follow-ups (from bl-838c review):

1. (knob) Make the timeouts config, not magic numbers — surface via the config layer (PartialConfig/ResolvedConfig) / flags like the rest of the gen settings, with the current values as row-defaults. Severable per AGENTS.md (policy lives in config, not core).

2. (gap) Add a configurable idle-read timeout on the body. Today connect + response-headers are bounded but the streaming body is intentionally unbounded so a long generation is never truncated — the cost is that a provider that sends headers then stalls mid-stream hangs bz forever. An idle / inter-chunk timeout (reset on each chunk) bounds the stall without capping total stream length. Note: ureq timeout_recv_body caps *total* body time, which is wrong for streaming — likely needs a read timeout on the socket or a wrapper around the into_reader() Read in the ChunkReader.

Both live in the bin (HttpTransport) + the config layer. Acceptable for v0.1 (the arch "caller spawns N bz and can kill them" model, §10), so not urgent.
