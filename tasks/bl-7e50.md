Follow-up to bl-410e. The raw reasoning channel (response.reasoning_text.{delta,done}) streams the reasoning item's content[] array (verified-org gated, carries content_index) and is currently dropped via the dispatch _ => [] arm — CR-R4 documented it as a deliberate v0.1 narrowing.

DECISION (user, 2026-06-16): implement the minimal route now. brazen should surface raw reasoning as Thinking, not only the summary.

SCOPE:
- Add dispatch arm in src/protocol/openai_responses/decode/mod.rs: "response.reasoning_text.delta" => delta(v, state, Delta::ThinkingDelta). It routes by part_key (output_index, content_index); for content_index 0 it lands in the Thinking block the reasoning item-add already opened — no new open logic. reasoning_text.done stays a no-op (output_item.done closes the block, same as the summary/text .done events).
- Hand-authored test in tests/responses_decode_errors.rs mirroring a_reasoning_item_opens_a_thinking_block_its_summary_delta_routes_in.
- Spec: add the reasoning_text.delta row to §3.4 table; rewrite CR-R4's 'deliberately unhandled' clause to 'handled (minimal)'.

CAVEATS (documented, not fixed):
- summary+raw coexistence: if a caller enables both reasoning.summary AND raw content, both route to pair (output_index,0) and concatenate into one Thinking block — lossless but redundant. A coexistence rule (raw-wins / summary-wins) is deferred to a real capture.
- multi-part raw (content_index > 0) mis-routes to an unopened pair and is dropped — the same limitation multi-part summary has.
