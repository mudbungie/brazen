Design task. Deliverable = amend specs/model-discovery.md (new caching section, reworking §1/§2/§5) + raise a CR to architecture.md §2 (the no-cache non-goal). Living doc, edited like code (AGENTS.md design-first). Implementation is a SEPARATE later ball, depends on bl-1189 (the list-models verb = the front door) landing.

GOAL: make imprecise-model resolution (`bz -m opus "hi"`) cheap on repeat by caching the resolved provider's model list, instead of todays fetch-fresh-discard inline probe (§5.2).

BEHAVIORS (user spec — note `bz`, not `bl`):
1. `bz "hi" -m opus`: consult the cache for the resolved provider. Miss -> run `bz list-models` to populate it, then re-read. Hit -> use it. Then select_model(list, seed) -> wire id (§4, unchanged).
2. A select_model that resolves from cache is used directly — no network.
3. Generation 404 (cached model stale upstream): re-run list-models (refresh cache), select_model again, retry the generation. `--fail-on-404` suppresses the refetch+retry (fail immediately) — the guard the user named against an unbounded loop.
4. FRONT DOOR: the cache is ONLY EVER WRITTEN by `bz list-models`. The generation path never hits the models endpoint itself; when it must (re)populate, it invokes `bz list-models` as a subprocess. One writer, one code path.

REVERSES COMMITTED ARCHITECTURE — resolve in the doc, do not paper over:
- architecture.md §2 Non-Goals: "Not stateful beyond ... (XDG config + credential/token storage). No history, no cache, no session files." and "No in-process fan-out. One request per process."
- model-discovery.md §1/§2: the imprecise probe is explicitly "**not** caching (the list is fetched fresh and discarded ... nothing is written to disk)."
  -> CR (arch.md changes FIRST, per the §7 CR discipline): add a third sanctioned XDG exception — a regenerable model-list cache — and name the front-door subprocess as a sanctioned control-path spawn.

OPEN DESIGN QUESTIONS TO SETTLE IN THE SPEC:
- Cache key = provider row name (cfg.provider.name, same key as CredStore store_key). Location = $XDG_CACHE_HOME/brazen/models/<provider>.json (cache is REGENERABLE -> CACHE_HOME, not the creds/STATE dir). Format = the {"models":[{id,default}]} shape `list-models --json` already emits (reuse, dont invent).
- Invalidation: NO TTL (a TTL is a flag-smell and contradicts the lazy intent); the sole invalidation is the 404 refetch. State as a non-goal.
- Infinite-loop REFRAME (AGENTS.md: dissolve the edge case with an invariant, not a flag): a 404 against a FRESHLY-fetched list is terminal — the model genuinely does not exist — so a refetch advances only if the list changed, bounding the retry to one by invariant. Then `--fail-on-404` is the extra knob ("dont refetch at all, fail on the first 404"), not the only guard. Decide which is load-bearing.
- Subprocess vs in-process front door: user wants a literal `bz list-models` subprocess (own-binary discovery via /proc/self/exe or argv[0], capture --json on stdout). Tension with minimalism + arch §2.3 "one request per process" — the alternative is calling the in-process fetch+cache-write fn (same single-writer goal, no spawn). Spec must justify the subprocess or adopt in-process.
- 404 seam: serve must distinguish a model-not-found 404 from other 4xx — carry the status (Frame.status / ErrorKind, the already-threaded fact, AGENTS.md), only a 404 triggers refetch.
- Concurrency: two `bz` writing the cache file -> atomic write (temp + rename) or accept last-writer-wins (cache is regenerable).
- `--fail-on-404` wiring: new CLI flag (config §); default OFF (refetch+retry once).
