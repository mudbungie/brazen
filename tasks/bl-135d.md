Design task. Deliverable = amend specs/model-discovery.md (rework §1/§2/§5 — the auto-probe is REPLACED by a read-only cache lookup, not extended) + a CR to architecture.md §2. Living doc, edited like code (AGENTS.md design-first). Implementation is a SEPARATE later ball.

DECISION (supersedes the earlier auto-listing draft): brazen NEVER lists automatically. `bz list-models` is the ONLY thing that hits the models endpoint and the ONLY writer of the cache; the generation path is READ-ONLY against the cache. Each invocation stays POSIX-clean — one network action, or zero — and arch §2 holds verbatim ("Not an agent ... no retry/backoff ... the caller orchestrates"). The cache-rule violation is ACCEPTED (a regenerable XDG file, nothing dirty); the auto-spawn / auto-retry is REJECTED (assumes too much for the user).

BEHAVIORS:
1. `bz list-models [--provider X]` — GET the resolved provider models endpoint, WRITE the cache. User-invoked. (Same verb as today; now it persists the list.)
2. `bz "hi" -m opus` (imprecise or absent model -> probe==true): READ the cache for the resolved provider.
   - cache PRESENT -> select_model(list, "opus") -> wire id (§4, unchanged) -> ONE generation round-trip.
   - cache ABSENT -> FAIL with a distinct, documented exit code + a message: run `bz list-models`. No auto-list, no spawn.
   - (probe==false, a fully-specified/owned/aliased model, is untouched: never reads the cache, one round-trip — unchanged.)
3. A cached model that resolves but 404s upstream (stale cache) -> FAIL (provider 4xx, exit 69) with a hint the list may be stale; the USER re-runs `bz list-models`. No auto-refetch, no retry loop.

WHAT THIS DELETES vs the prior draft / the in-flight impl:
- The auto-probe (model-discovery §5.2 `fetch_models` in serve) — REMOVED; generation reads the cache instead of prepending a live list. NB: bl-1189 is building that auto-probe right now — this supersedes that part (the `list-models` verb + `select_model` + `Model`/`decode_models` it also builds are KEPT and reused).
- The `--fail-on-404` flag — GONE (failing IS the behavior; nothing auto-retries to suppress).
- The front-door SUBPROCESS — GONE (generation never lists, so nothing shells out). "Front door" survives only as: list-models is the sole cache writer.
- The infinite loop — GONE by construction.

COST MODEL (restored, better than the probe): fully-specified = 1 round-trip; imprecise + warm cache = 1 round-trip (was 2 with the probe); imprecise + cold cache = 0 network, exit code. `bz list-models` = 1 round-trip, a separate invocation. NO path does two round-trips.

CACHE: key = provider row name (cfg.provider.name, same key as CredStore store_key). Location = $XDG_CACHE_HOME/brazen/models/<provider>.json (REGENERABLE -> CACHE_HOME, not the creds/STATE dir). Format = the {"models":[{id,default}]} shape `list-models --json` already emits (reuse, dont invent). No TTL (sole invalidation is the user re-running list-models). Atomic write (temp + rename) so concurrent `bz` dont corrupt it.

CR to architecture.md §2 (changes FIRST, per the §7 CR discipline): add a regenerable model-list cache as a sanctioned XDG exception alongside config + creds. Note that §2 "no retry/backoff, caller orchestrates" is now HONORED, not bent: a wrapper maps the cold-cache exit code -> `bz list-models` -> retry; brazen exposes the condition and never acts.

OPEN QUESTIONS:
- Exact exit code for the cold-cache miss: a DISTINCT documented code so a wrapper can orchestrate the list (recommend a dedicated value — e.g. EX_NOINPUT 66 "no model cache; run bz list-models" — over reusing Config 78, which already means provider/model-resolution failure). Decide.
- Stale-404 hint: emit "list may be stale" only when the model came FROM the cache (vs a user-typed full id) — needs the came-from-cache fact carried — or just always hint on a 404. Decide.
