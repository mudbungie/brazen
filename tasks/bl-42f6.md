## Brazen server-tools ball ‚Äî IMPLEMENTATION SPEC (resolves CR-4, OPAQUE passthrough)

### 0. Scope & invariant
Resolve CR-4 ("server tools deferred") as **opaque verbatim passthrough** of Anthropic server-tool ENABLEMENT and RESULT blocks. **No** canonical `WebSearch`/citation normalization, **no** new `Usage` field. The model gains: two additive `Content` variants, two additive `ContentKind` variants, and two additive `Tool` fields (`kind`/`config`) + an `input_schema` default. Everything round-trips as opaque `serde_json::Value`.

**v=1 forward-compat note (load-bearing):** these are *additive* kinds/fields within `EVENT_SCHEMA_VERSION = 1`. Per `event.rs` module doc (lines 6‚Äì13): "Within a fixed `EVENT_SCHEMA_VERSION` the vocabulary only GROWS‚Ä¶ `v` bumps ONLY for a removal, rename, or semantic change ‚Äî never for an addition." So **no version bump** ‚Äî the additive empty-set/`v=1` rule the arch predicted holds, and the spec must state this explicitly in the doc edits.

**Betas are the caller's job:** brazen ships **no vendor policy** on `anthropic-beta` headers. GA server tools (web_search, web_fetch, code_execution, tool_search) need no header; beta tools (computer use, advisor) do. The caller adds any required header via the existing config-row `beta_headers` mechanism. Do **not** bake tool-version‚Üíbeta tables into brazen.

**Wire-tag caveat (the central footgun):** the result block's Anthropic wire `type` is `web_search_tool_result`, NOT the snake_case of a generic `ServerToolResult` (`server_tool_result`). The variant is generically named but **hard-wired to web_search** via `#[serde(rename = "web_search_tool_result")]`. Other server-tool result tags (`code_execution_tool_result`, `bash_code_execution_tool_result`, etc.) will NOT round-trip ‚Äî that is in-scope per CR-4 (web_search only). If broader generality is later wanted it needs a `wire_type: String` field instead; flag, do not build.

**CR-4 naming collision (do NOT cross the streams):** `anthropic-messages.md ¬ß6 "CR-4"` (server tools ‚Äî the one we resolve) is a *different* change request from the already-closed serde-external-tagging "CR-4" referenced at `architecture.md` L100/L164/L188/L203 and `openai-chat-mapping.md` L517‚Äì519. Touch ONLY the server-tool deferral text. Leave every serde-tagging CR-4 comment alone.
