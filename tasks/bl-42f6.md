# Resolve CR-4: first-class Anthropic server-tool support as opaque verbatim passthrough (open-set `*_tool_result` family; web_search fixtures), no canonical normalization

## Adopted resolutions (open decisions — already decided, do NOT re-litigate)
- **Google `parts_value`: DROP** the two new Content variants (fold into the existing `Thinking|RedactedThinking` empty-set drop arm) — required to compile; consistent with the existing empty-set drop. (NOT reject.)
- **AMENDED 2026-07-01 (user ruling — supersedes the earlier "hard-wired `web_search_tool_result` / do NOT add wire_type" resolution): the result wire tag is CARRIED AS DATA.** `ServerToolResult` gains `kind: String` — the verbatim wire `type` (`web_search_tool_result`, `code_execution_tool_result`, …). Decode keys on the tag SUFFIX (`ends_with("_tool_result")` && tag != `"tool_result"`); encode re-emits `kind` verbatim. The whole `*_tool_result` family round-trips with ZERO per-tool knowledge — the open-set rule (`type` is opaque like `model`) applied to result blocks. First-class FIXTURES stay web_search-only (that is the verified golden); the suffix rule is what makes the rest round-trip instead of corrupting replay.
- **AMENDED 2026-07-01 (user ruling): `Tool` becomes a TWO-VARIANT ENUM**, not additive optional fields: `Custom { name, description, input_schema }` | `Provider { kind /* wire "type" */, name, config }`, hand-rolled serde keyed on the presence of the `"type"` key (NOT `#[serde(untagged)]` — unusable errors). Illegal states (both/neither) become unrepresentable and the flatten-ordering footgun the earlier draft had to test around disappears. Wire-compatible: a tool object with no `type` key decodes `Custom` exactly as today.
- **`usage.server_tool_use.web_search_requests` stays deferred** — no canonical `Usage` field; rides `provider_detail`. The content BLOCKS resolve; the usage COUNTER does not. Keep it documented as the residual.
- **`event.rs` SPLIT is mandatory** (299/300 lines) → move `tag_of`/`str_at` + hand-rolled ContentKind/Delta/FinishReason serde to NEW `src/canonical/event_serde.rs` (mirrors request.rs/request_de.rs). Verify both halves ≤300 with `wc -l` (aim <200 — 200 is the thickness indicator).
- **Fixtures:** craft `anthropic_messages_web_search.sse` + `anthropic_messages_nonstream_web_search.json` from the verified docs shapes in the spec (incl. the error-object content form), deterministic under one-byte rechunk. Live capture is preferred if feasible but the crafted golden is acceptable.
- **v=1 contract:** these are ADDITIVE kinds — NO `EVENT_SCHEMA_VERSION` bump. State this in the doc edits.
- **Do NOT touch the unrelated serde-tagging "CR-4"** comments (architecture.md L100/L164/L188/L203, openai-chat-mapping.md L517-519) — different change request from the server-tool CR-4.

## Brazen server-tools ball — IMPLEMENTATION SPEC (resolves CR-4, OPAQUE passthrough)

### 0. Scope & invariant
Resolve CR-4 ("server tools deferred") as **opaque verbatim passthrough** of Anthropic server-tool ENABLEMENT and RESULT blocks. **No** canonical `WebSearch`/citation normalization, **no** new `Usage` field. The model gains: two additive `Content` variants, two additive `ContentKind` variants, and `Tool` re-modeled as the two-variant enum above. Everything round-trips as opaque `serde_json::Value` + the verbatim wire tag.

**brazen executes nothing.** Client tools (custom AND Anthropic-schema typed like bash/computer) are executed by the harness via the existing `ToolUse`→`ToolResult` flow; server tools were already executed by the provider before the bytes arrived. brazen's three jobs at this boundary: TRANSLATE (project declarations), CLASSIFY (server results are done, never pending — the litellm-class failure is a classification failure), ROUND-TRIP (verbatim, or replay 400s).

**v=1 forward-compat note (load-bearing):** these are *additive* kinds/variants within `EVENT_SCHEMA_VERSION = 1`. Per `event.rs` module doc (lines 6–13): "Within a fixed `EVENT_SCHEMA_VERSION` the vocabulary only GROWS… `v` bumps ONLY for a removal, rename, or semantic change — never for an addition." So **no version bump** — the additive empty-set/`v=1` rule the arch predicted holds (unmapped kinds already rode `ContentKind::Other(Value)`), and the spec must state this explicitly in the doc edits.

**Betas are the caller's job:** brazen ships **no** vendor policy on `anthropic-beta` headers. GA server tools (web_search, web_fetch, code_execution, tool_search) need no header; beta tools do. The caller adds any required header via the existing config-row `beta_headers` mechanism. Do **not** bake tool-version→beta tables into brazen.

**CR-4 naming collision (do NOT cross the streams):** `anthropic-messages.md §6 "CR-4"` (server tools — the one we resolve) is a *different* change request from the already-closed serde-external-tagging "CR-4" referenced at `architecture.md` L100/L164/L188/L203 and `openai-chat-mapping.md` L517–519. Touch ONLY the server-tool deferral text. Leave every serde-tagging CR-4 comment alone.

```rust
#[derive(Clone, Debug, PartialEq)] // hand-rolled serde in request_de, keyed on `type`
pub enum Tool {
    /// Caller-defined: brazen understands the STRUCTURE (name, description, JSON
    /// Schema) and PROJECTS it across dialects (Anthropic `input_schema` vs OpenAI
    /// `function.parameters`) — that projection is the whole value.
    Custom {
        name: String,
        description: Option<String>,
        input_schema: Value,
    },
    /// Provider-typed (Anthropic-schema client tools bash/computer/… AND server
    /// tools web_search/…): opaque `kind` (wire `type`, e.g. `web_search_20250305`)
    /// + config, passed through to the routed provider verbatim. brazen has no
    /// opinion on `kind`; the provider is the authority (a bad one → provider 400).
    Provider {
        kind: String,
        name: String,
        config: Map<String, Value>,
    },
}
```

#### 1.2 `src/canonical/request_de.rs` — manual serde, BOTH directions
**Content.** `ServerToolUse` joins the existing `Tagged` mirrors (tag `server_tool_use`, snake_case matches — no rename). `ServerToolResult` CANNOT be a fixed tagged variant (its tag is dynamic):
- Serialize: emit the map `{"type": kind, "tool_use_id": …, "content": …}` (hand-rolled arm — `serde_json::json!` or a serialize_map).
- Deserialize: BEFORE the fixed-`Tagged` dispatch, intercept objects whose `type` tag satisfies `tag.ends_with("_tool_result") && tag != "tool_result"` → `Content::ServerToolResult { kind: tag, tool_use_id, content }`. The `!= "tool_result"` guard keeps the client `ToolResult` decode untouched.

**Tool.** Hand-rolled `Serialize`/`Deserialize` keyed on the presence of `"type"`:
- Deserialize: object WITH a `type` key → `Provider { kind: type, name, config: every remaining key except type/name }`; WITHOUT → `Custom` (today's shape; absent `description` stays `None`, absent `input_schema` is an input error exactly as the current derive made it — preserve current strictness for `Custom`).
- Serialize: `Custom` → `{name, description?, input_schema}` (omit `description` when `None` — matches today's wire); `Provider` → `{"type": kind, "name": name, ...config}` (config flattened; no `input_schema`, no `description`).
Round-trip law: `config` captures EVERY key except `type`/`name`, so unknown provider config (max_uses, allowed_domains, user_location, …) survives verbatim.

Line budget: `request_de.rs` grows — if it crosses 300 (or lands thick >200), split a sibling (`request_de_tool.rs`); check `wc -l`.
