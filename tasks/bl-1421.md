Motivation: lernie (ARCHITECTURE v0.4 §4.2) keeps a hand-maintained models.yaml with context_window/capabilities because bz --list-models returns ids only. Several providers DO report metadata on their list/show endpoints — Google generative-ai models.list returns inputTokenLimit/outputTokenLimit; Ollama /api/show reports model details; OpenAI/Anthropic mostly do not. Where the provider is authoritative for a fact, a harness should be able to derive it instead of hand-mirroring it (single source of truth); hand config remains only where the fact genuinely is not served (empty-set rule).

Design first: spec in model-discovery.md. Additive and Option-shaped throughout — absent metadata stays None, never fabricated (the Usage zero-vs-unknown principle); the --json output shape ({"models":[...]}) gains optional per-model fields; the per-provider model cache schema extends additively (a cache written by an older bz still reads — mirror the v=1 grows-only discipline); text mode (ids one per line) unchanged. The models-shape DATA table (model-discovery.md §3/§3.1) grows optional key paths per protocol; the [provider.models] row override (§3.2) should be able to name them too. No new flags, no new round-trips — same one GET.
