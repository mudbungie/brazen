RULED + IMPLEMENTED (user sign-off 2026-07-17), awaiting close.

RULING: greedy first-match over a priority list. NOT the tiered alias>prefix rule this ball proposed, and not a specificity comparator — the config IS a priority list, routing takes the first owning row, AmbiguousModel is RETIRED (a total order always has a first element; no tie to surface). Precedent: model-discovery.md 4 already ruled the same shape ('List order is authoritative... No ambiguity error — the order IS the tiebreak'); routing was the outlier.

STRUCTURE (changed under fable review): PartialConfig.providers is now Vec<(String, PartialProvider)> — the row table IS the priority list. Rejected: provider_order:Vec<String> beside a BTreeMap (stores the name set twice; policed invariant; two merge fns that must agree — and as specced it leaked priority onto the wire as an authorable TOML key). Order is now intrinsic: greedy-first IS .find(row_owns), zero-config default IS .first(), default_provider field DELETED, no order-merge fn, dump emits rows in list order (replacing the sort_by_key hack at dump.rs:128).

NEW VALIDATION (fable catch): model_prefixes = [""] is BadValue/78 — "x".starts_with("") is true, so an empty prefix element silently owns EVERY model; the retired ambiguity rule used to catch it by accident. model_prefixes = [] (empty list) stays legal.

DOCUMENTED FOOT-GUN (fable catch): per-field merge + priority means a user row redeclaring a defaults row (e.g. only a body_defaults tweak) keeps the defaults' claims but takes YOUR position — so a tweak above an alias row silently re-arms gpt- and kills the masquerade. Stated in config.md 3.2, warned in ingress.md 6, pinned by test.

HONEST COST (in architecture.md 4.3): the pick is not surfaced. --dump-config shows the ORDER, not which row won for model X. No --why-this-row added (a routing-explainer verb is the mechanism this ruling avoids). Mitigation is --provider.
