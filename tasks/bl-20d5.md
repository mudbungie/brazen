## Why
`drive()` only decodes a *streamed* 2xx (SSE/NDJSON framing); every concrete protocol frames a stream, and there is no non-stream-2xx fold (architecture §3.2's 'folded stream' is unimplemented for 2xx). A request that doesn't opt into streaming yields a single JSON the decoder can't parse → `premature upstream EOF`. [bl-ad92] made `--stream` reach the wire; this makes streaming the *default* so bare `bz <prompt>` works against SSE providers — what brazen should handle implicitly.

## Change
In `serve` (post-dispatch, canonical path, after fill_absent / before encode): `req.stream.get_or_insert(true)`. brazen owns the round-trip — serve both requests streaming and SSE-decodes the 2xx in `drive` — so the default lives there. A request/config that sets `stream` still wins (incl. explicit `false`). Pure `encode` stays absent→false (conservative projection for direct lib callers); serve applies the wire default.

## Done when
- `bz <prompt>` (no --stream) against anthropic streams cleanly (live-verify).
- Tests pin: None→stream:true through serve; Some(false)→stays false.
- make check green; specs/config.md §4.2 + architecture.md note the serve-applied streaming default.

## Note (out of scope)
Explicit `stream:false` on an SSE provider still breaks (drive has no non-stream-2xx fold). Separate concern — file if we want stream:false supported.
