openai_responses decode (landed in bl-2f13) keys the canonical block index off the wire `output_index` and assumes ONE content block per output item. A single `message` output item that streams multiple content parts (distinct `content_index` values, e.g. text + a second text/refusal part) would route both to the same canonical index -> the second content_part.added overwrites the first's open block, deltas misroute, and the cross-check invariant (one start per index) breaks.

Real but rare for v0.1 (the common Responses shape is one output_text part per message item; function_call items are their own output item). No fixture exercises it; the code does not handle it.

Fix: incorporate content_index into the canonical index assignment (e.g. a (output_index, content_index) -> canonical-index map, the way Anthropic's wire index already is the canonical index). Add a multi-part fixture. Also reconcile providers.md §3.4 (currently says 'the canonical index is the wire output_index'). Low urgency.
