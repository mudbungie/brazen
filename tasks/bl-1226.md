0.0.1 published live to crates.io (workflow_dispatch). But release-binaries.yml never ran (zero runs): release-plz creates the GitHub Release with the default GITHUB_TOKEN, and GitHub deliberately does NOT let token-created releases trigger another workflow (anti-recursion). So 'on: release: published' is a dead trigger here without a PAT.

## Fix (no PAT; truly hands-off)
- Delete release-binaries.yml; fold its 7-target matrix into release-plz.yml as a new job release-binaries that needs: release-plz-release and runs in the SAME run — so there is no cross-workflow cascade to be blocked.
- Gate: if needs.release-plz-release.outputs.releases_created == 'true' (a real release just happened) OR (workflow_dispatch && inputs.binaries_tag != '') (manual backfill).
- release-plz-release: add id: release + outputs.releases_created mapped from the step.
- workflow_dispatch input binaries_tag (default '') to build/upload binaries for an existing tag.
- Binaries job: checkout fetch-depth 0; resolve tag = inputs.binaries_tag else git describe --tags --abbrev=0; git checkout that tag (build the released code); taiki-e/upload-rust-binary-action with ref: refs/tags/<tag> (upload target) + token GITHUB_TOKEN (asset upload is allowed; only NEW-workflow triggering is blocked).

## After landing
Dispatch with binaries_tag=v0.0.1 to backfill 0.0.1's binaries onto its existing Release. Future releases auto-build binaries in the same green-gated run.

## Close gate
YAML parses; make check green.
