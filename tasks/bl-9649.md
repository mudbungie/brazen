Found via bl-2869. A TOP-LEVEL provider row key (e.g. unsupported_body_keys) typed under [provider.oauth] is silently dropped — no error, --dump-config shows it absent — so the strip never fires and the operator still 400s. Same footgun as a typo'd oauth key. ProviderRow has #[serde(deny_unknown_fields)] (typo -> MalformedFile, config §2.3); OAuthConfig (and RedirectSpec) do not. Add deny_unknown_fields to OAuthConfig for parity. The auth.md §10.7 recipe placement bug this caused is already fixed in bl-2869; this hardens the parser so the next misplacement is caught, not swallowed.
