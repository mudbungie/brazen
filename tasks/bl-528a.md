## Problem
The `ollama` provider row is `auth = "bearer"`, so `bz` demands a credential (`MissingCreds`/77) even though Ollama ignores the Authorization header. `data/defaults.toml` claims 'bearer is set only if a key exists, else the missing key is tolerated' — that is fiction: `StaticSecretAuth`/`resolved_secret` (src/auth/mod.rs:105) errors unconditionally. There is no no-auth mode; keyless-local was shoehorned into bearer.

api-key is not universally required (oauth2 rows already bypass it via a separate impl). The fix is a real no-auth capability, not tolerating a missing key on bearer (which would silently downgrade a forgotten OpenAI key from a clean 77 to a provider 401).

## Fix (auth = "none", confirmed correct by user)
- `AuthId::None` (`#[serde(rename = "none")]`) in src/config/provider.rs; registry arm `AuthId::None => &NoAuth`.
- `NoAuth` Auth impl: no-op (sets no header, reads no credential).
- `Provider.api_header: Option<HeaderSpec>` (#[serde(default)]) — a no-auth row carries no auth header. Mirror the `oauth` resolve-invariant pattern: in resolve::complete, require api_header iff `auth != None` (else IncompleteProvider/78).
- `ProviderCtx.api_header: Option<&HeaderSpec>`; serve.rs builds `.as_ref()`. StaticSecretAuth + OAuth2Auth(refresh.rs:69) take the header via a shared `require_header` helper that returns a defensive Config error (→78) on None — exercised by a direct unit test, exactly like `oauth_row_misconfigured`.
- data/defaults.toml: ollama row → `auth = "none"`, drop `api_header`, fix the comment.

## Done when
- `bz --provider ollama --model <m> "hi"` works with NO --api-key (and --api-key still accepted, ignored).
- 100% line coverage holds; new tests: NoAuth no-op, resolve requires/omits api_header by auth mode, require_header None defensive path.
- ollama_smoke.rs (bl-76fd) can drop --api-key dummy; scripts/smoke.sh ollama row (no key) becomes correct.
