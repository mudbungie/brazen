USER ASK (2026-07-19, verbatim): "if I type: bz --model 5.5, and my default is
anthropic, where there is no model that matches 5.5, it should fail out and use
the second provider, openai, where there absolutely is one."

SHIPPED: routing ownership gains a SECOND TIER. A row owns a model it CLAIMS
(model_aliases / model_prefixes -- today's rule, untouched) OR, when NO row
claims it, one its CACHED model list PLACES under select_model's own rule
(exact id, else first case-insensitive substring). No new matching rule, no
flag, no config key, no network in routing. Delete the cache and behaviour
degrades exactly to the old claims-only rule.

  src/config/resolve/mod.rs -- route() reads the priority list ONCE PER TIER:
    .find(row_claims) .or_else(|| .find(cache_places))
  cache_places() == matches!(select_model(cache.get(name), model, name),
                             Ok((_, Provenance::Cached)))

RULINGS ON THE FOUR OPEN QUESTIONS (all as recommended; no deviations):

1. TIER PRECEDENCE -- WHOLE-LIST, two passes. Every row's CLAIM is checked
   before ANY row's CACHE. A claim on a later row therefore beats a cache hit
   on an earlier one. Consequence: a stale/learned cache entry can only ever
   win a model that NO row claims at all -- exactly the gap the tier fills.
   Test: config_fallthrough::a_claim_on_a_later_row_beats_a_cache_hit_on_an_earlier_one

2. COLD CACHE -- NO PROBING. A row with no cache file cannot win tier 2.
   Network I/O in config resolution is refused; the limitation is documented in
   architecture.md 4.3.2 and the NoProvider message names the remedy.

3. LEARN-ON-SUCCESS (src/run/generate.rs 2xx append) -- CANNOT HIJACK. Ruling 1
   subsumes it: an appended id is only read in tier 2, after every row's claims
   have been checked, so a one-off `--provider anthropic --model gpt-5.5` that
   returned 2xx can never divert the gpt- family from the row that CLAIMS it.
   No separate guard was added. Recorded in model-discovery.md 5.4.

4. ERROR TEXT -- updated: "no provider resolved: name one with --provider, or
   use a model a provider owns (a known alias, a model-id family, or an id in
   some provider's cached model list -- run `bz --list-models`)".

EXTRA DISSOLVE (not in the brief): an EMPTY routing model is ABSENCE, not a seed
that owns nothing. `--model ""` / `model = ""` now resolves like the bare
`bz "q"` (first declared row, empty seed) instead of NoProvider. Without this
both tiers get asked to place "" and select_model reads an empty seed as
"the default", which would have made row 1 win any non-empty cache.
