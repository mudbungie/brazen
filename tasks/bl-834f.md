USER ASK (2026-07-19, verbatim): "if I type: bz --model 5.5, and my default is
anthropic, where there is no model that matches 5.5, it should fail out and use
the second provider, openai, where there absolutely is one. right now, if I try
bz --model 5.5 '...' without --provider codex, it errors."

REPRO (real binary, main @ 8f8500d):
    $ bz --model 5.5 "hi"
    no provider resolved: name one with --provider, or use a model a provider
    owns (a known alias or model-id family)

DIAGNOSIS: the matching the user wants ALREADY EXISTS, one stage too late.

  Stage A routing (src/config/resolve/mod.rs:144-154, `row_owns`):
      aliased || prefixed
      alias  = model_aliases.contains_key(model)   -- exact key hit
      prefix = model.starts_with(p)                -- starts_with only
    "5.5" hits neither for any row -> NoProvider (exit 78) at :127-129.

  Stage B selection (src/canonical/model.rs:85-98, `select_model`):
      exact id, else FIRST cached id containing seed case-insensitively,
      else Verbatim (total; never errors on no-match)
    "5.5" CONTAINS-matches "gpt-5.5". It does NOT match "claude-sonnet-5-5"
    ("5.5" with a dot is not a substring of "5-5") -- precisely the user's
    anthropic-misses / openai-hits scenario.

  Stage B reads $XDG_CACHE_HOME/brazen/models/<provider>.json
  (src/native/cache.rs:19-71). Routing never consults it.

PROPOSED CHANGE (attack it before building): extend row ownership to a second
tier -- a row owns the model if its CLAIMS own it (today's rule), OR its cached
model list contains a select_model-style match. Walk rows in priority order,
first owner wins (the bl-942d greedy rule, unchanged). Only when no row owns it
by either tier is it NoProvider.

  - No new mechanism: reuses the existing cache + existing substring semantics.
  - No network in the hot path: the cache is local JSON, forgiving-read
    (cache.rs:56-63 returns None on any corruption).
  - No new config surface, no new flag. Severable: delete the cache, behaviour
    degrades to today's claims-only routing.

OPEN QUESTIONS TO SETTLE IN THE BALL, NOT ASSUME:
  1. Tier precedence. Must a CLAIM beat a CACHE hit on a LOWER row? Proposed
     yes: two passes, all-rows-by-claim then all-rows-by-cache. Otherwise a
     stale cache entry on row 1 silently steals a model row 2 explicitly
     claims by prefix. This is a real precedence ruling -- write it down.
  2. Cold cache. A provider with no cache file can never win tier 2, so
     fall-through is invisible until `bz --list-models` has run for that
     provider. Is that acceptable, or should a cold row be probed? Probing
     means network-in-routing -- proposed NO, document the limitation.
  3. Does learn-on-success (src/run/generate.rs:113-122, appends a Verbatim id
     to the cache tail on 2xx) now let a one-off explicit run permanently
     re-route a family? Check the interaction.
  4. Error text at :127-129 must change -- it currently says "a known alias or
     model-id family", which will no longer be the whole rule.
