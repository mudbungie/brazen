Motivation: lernie (the git-backed agent harness at ~/dev/lernie) is adopting bz as its provider adapter (lernie ARCHITECTURE v0.4 §4.4). Its context assembler enforces per-role token budgets (manifest budget_tokens) and compaction triggers, which today run on estimates. Provider-accurate counting is wire knowledge — brazen's domain, not the harness's.

Design first (AGENTS.md): spec before impl. Shape per architecture.md §5.10.1 — control operations are FLAGS in the control-short-circuit family, never verbs, so the bare-prompt namespace never shrinks. Proposed: bz --count-tokens reads a canonical request on stdin (same parse path as the data plane), resolves provider/model exactly as the data plane does, performs one round-trip to the provider's count endpoint where one exists (Anthropic POST /v1/messages/count_tokens), and emits {"input_tokens": N} under --json / bare N under text. Providers without a count endpoint: rule the arm explicitly in the spec — Config error (decline-illegal-operations) vs a documented estimate; leaning error + the estimate stays the caller's fallback, since a fabricated count is a lie (the Usage Option-not-zero principle applied to counting).

Spec surfaces to touch: architecture.md §5.10.1 committed control shapes + mutual-exclusion rules, §8 exit codes (reuses existing classes), anthropic-messages.md (count endpoint mapping), model-discovery.md precedent for the one-GET control-op pattern. Mutually exclusive with the other control ops; probes still win. One round-trip, no retry, no cache write.
