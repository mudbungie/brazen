Raised while reviewing the wave-3 close (454ebfa), NOT a directive — could be a non-issue. Since bl-8ec6, the serve path owns dialect selection (POST /v1/messages -> anthropic_messages, POST /v1/chat/completions -> openai_chat); the required [ingress].dialect field's remaining serve-side job is choosing the error envelope for surfaces no path owns (unknown routes/404, malformed HTTP, shared GET /v1/models). Question: does a required config field earn its keep decorating errors on routes that do not exist? Options to weigh, none pre-chosen: (a) keep as-is — the field still governs the one-shot --in/--out filter path, so maybe serve is just borrowing it honestly; (b) make it optional with a default for the routeless surfaces; (c) drop it from serve semantics entirely and pick a fixed envelope for routeless errors. Check what the filter path actually needs before concluding. Deliverable: a ruling recorded in specs/ingress.md (amend the route/dialect sections either way, even if the answer is keep).
