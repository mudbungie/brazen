## Context
`src/run/serve.rs:112` does `req.stream = Some(true)` unconditionally on the canonical path, SILENTLY reverting any explicit `stream:false` (request / config / `body_defaults`). Wrong semantic: a user flag is silently ignored, not honored. (Also the root of the bl-cc84 misdiagnosis — forced `stream:true` masquerading as "codex dropped its mandate".)

USER DECISION: never silently ignore+revert a flag — EITHER ERROR OR HONOR; and brazen should SUPPORT EITHER streaming and non-streaming, because faithfully speaking each provider's wire dialect (incl. its non-stream response shape) is the whole point of brazen. The only open policy question — the DEFAULT when unset — resolves provider-by-provider from the row's `body_defaults`, with brazen's stream-native global fallback of `true`.

Today the whole response pipeline only decodes a FRAMED stream (`drive` -> `stream()` over SSE/NDJSON); there is NO non-stream-2xx fold. So honoring `stream:false` needs a per-protocol decode of the single aggregate JSON body.

## Design — single source of truth: explode -> replay, don't re-parse
A non-stream body is the AGGREGATE of what the stream emits. Do NOT write a second parser per protocol. Each protocol's non-stream decode reconstructs the synthetic event sequence the stream would have produced and replays it through the protocol's EXISTING `event`/`line`/`chunk` + terminal helpers. e.g. an `openai_responses` non-stream body IS the same `response` object the streaming `response.completed` event wraps as `v["response"]`; `terminal::{completed,usage,completed_finish}` are already pure fns of it, reused verbatim. Structureless protocols (`openai`, `google_genai`, `ollama_chat`) reuse `synth::{open_text,next_index,drain}` exactly as their streaming decoders do. (Per-protocol size est. 25-55 lines.)

## The seam (one new trait method)
`Protocol::decode_full(&self, body: &[u8], state) -> Result<Vec<Event>, CanonicalError>` (`src/protocol/mod.rs`): decode a COMPLETE non-stream 2xx body -> canonical events (message_start .. finish; never End — run owns it, like `decode`). DEFAULT impl ERRORS (`ErrorKind::Config`, exit 78, clear in-band msg "non-streaming not yet supported for this provider; omit `stream` or set it true") so protocols fill in incrementally with NO silent revert (honor-or-error interim). The default is covered while >=1 protocol still relies on it; the LAST protocol ball removes the default, making `decode_full` REQUIRED (no dead/uncovered code under the 100% gate).

## Routing (`src/run/respond.rs` `drive`, `src/run/serve.rs`)
- Capture `streamed = req.stream.unwrap_or(true)` in serve's Canonical arm (the wire intent encode emits); thread into `drive(..., streamed)`.
- New `drive` branch — only Canonical (non-raw), 2xx, `!streamed`: `whole_body_success(...)` drains the body (reuse `drain`) and calls `proto.decode_full`; no premature-EOF check (body is complete).
- `--raw` stays on `stream()`/Identity unchanged.

## Default policy (severable — policy in the row, not core)
- Remove `serve.rs:112` override.
- `fill_absent` (`resolved.rs:75`): `req.stream.or(cfg.stream)` already = request > flag/env/file > body_defaults (fold at `resolve.rs:30`). Add brazen's stream-native global fallback `.or(Some(true))`. A provider that works better non-streamed pins `body_defaults = { stream = false }` (config, not code) — severability test passes.
- Add `--no-stream` to `src/cli.rs` (`cfg.stream = Some(false)`); only `--stream` exists today. `BRAZEN_STREAM=false` already works.
