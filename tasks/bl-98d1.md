Spec providers.md §3.4 row 152 says:

| `response.reasoning_summary_text.delta` | `ContentDelta{index, ThinkingDelta(delta)}` (block opened with `ContentStart{Thinking {}}` on the matching `reasoning` item add) | — |

But the code does NOT open a Thinking block. In src/protocol/openai_responses/decode/mod.rs:

- `item_added` opens a block ONLY for `item.type=="function_call"` (synthesizes `ContentStart{ToolUse}`); a `reasoning` item add returns `vec![]` and opens nothing.
- `part_added` opens a block ONLY for `part.type=="output_text"`, as `ContentKind::Text {}` — never `Thinking`.

So `response.reasoning_summary_text.delta` (dispatched to `delta(.., Delta::ThinkingDelta)`) routes by its `(output_index, content_index)` pair to whatever block that pair opened:
- if no matching block opened (the reasoning summary part is not an `output_text` part) → the thinking delta is DROPPED (routes nowhere).
- if a reasoning summary somehow rode an `output_text` part → the `ThinkingDelta` lands in a block whose kind is `Text`, a kind/delta mismatch.

Either way, reasoning-summary streaming from Responses reasoning models (o-series, gpt-5 reasoning) is not actually surfaced as a canonical Thinking block, contradicting the spec.

Resolution options (design first):
1. Open a `Thinking` block on the matching `reasoning` output item / its reasoning summary content_part (mirror the function_call identity-before-content path in `item_added`/`part_added`), so the pair maps to a `Thinking` block before its first `reasoning_summary_text.delta`. Reconcile the spec row with whatever the real wire shape is.
2. Or, if reasoning replay/streaming is deliberately out of scope for v0.1 (cf. CR-R3, which drops `Content::Thinking` on re-send), correct spec §3.4 row 152 to say the reasoning-summary delta is dropped in v0.1, and remove the misleading "block opened with ContentStart{Thinking}" claim.

NOTE: pinning down the real Responses reasoning_summary wire shape needs a live/recorded fixture (the existing fixtures are hand-authored; see bl-aba5). The current test `reasoning_summary_delta_routes_to_an_open_block` in tests/responses_decode_errors.rs opens an `output_text` content_part and routes a ThinkingDelta into that Text block — i.e. it asserts the mismatched behavior, so it should be revisited as part of the fix.
