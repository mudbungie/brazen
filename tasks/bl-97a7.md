Arch-review finding (2026-07-07). google_genai/encode/contents.rs:75 maps ImageSource::Url -> {fileData:{fileUri:url}}. Ground truth: Gemini fileData.fileUri accepts only Google-hosted URIs (Files API / GCS), typically with mimeType, and cannot fetch an ordinary https://...png — unlike OpenAI chat/responses which pass web URLs through natively. So a canonical Image{Url} routed to google fails at the provider with a confusing 400, while the encode implies support.

Decide per the architecture's own rule (architecture.md §3.1: a wire slot that cannot express the content REJECTS at encode with ParseInput/64 — a documented runtime degradation, never a silent mistranslation): (a) reject Image{Url} at google encode with a message naming the limitation (caller downloads and re-sends Base64 — brazen never adds round-trips, §2), while KEEPING the fileData emission for URIs that genuinely are Google-file/GCS URIs (detectable prefix? spec-argue whether prefix-sniffing is acceptable or the reject should be total), or (b) keep emitting and document the narrowing as caller-beware. Lean (a)-total-reject unless the prefix rule is clean: generativelanguage.googleapis.com/gs:// prefixes are unambiguous. providers.md §4.3 image row + fixture + CHANGELOG. Small lane: google encode only.
