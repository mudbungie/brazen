Adopt Option B (POSIX/Unix filter idiom): a positional prompt makes stdin irrelevant — read_request(Some(prompt), _) builds the request from argv and NEVER reads the reader. An unread pipe is the writer's concern (EPIPE/SIGPIPE), exactly like `head`. This removes the only reason read_request touched stdin when a prompt was present, which was the §5.5 'positional + non-empty stdin -> exit 64' rule; that rule forced a blocking read on an interactive tty and is not POSIX-mandated (POSIX is silent; the idiom is 'don't read input you don't need').

Changes:
- read_request: Some(prompt) -> ignore reader entirely (no read_to_end, no 64).
- specs/architecture.md §5.5, Non-Goals §2, Resolved Decision #7: drop the mutual-exclusion/exit-64 clause; state positional ignores stdin and an unread pipe breaks upstream via SIGPIPE.
- Tests: drop the both-present->64 tests (pipeline_input + run_modes) and the prompt+unreadable-stdin->64 test; add a test proving a positional prompt ignores a non-empty/failing stdin reader.
- Keep 100% line coverage, files <300 lines.
