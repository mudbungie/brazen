## Defect

The lazy model-discovery probe (bl-ea58/bl-67e2) fires the list-models GET on EVERY generation for any provider row that has neither `model_prefixes` nor a matching `model_aliases` entry — even when `--model` is a fully-qualified, exact wire id. This breaks the canonical generation path for the `openai-chatgpt` (Codex) row and the shipped `openai-responses` default row.

## Root cause

`src/config/resolve.rs:37`:
```
let probe = routing_model.is_none_or(|m| !row_owns(&partial, m));
```
`row_owns` (resolve.rs:119) returns true only if the model is in `model_aliases` OR matches a `model_prefixes` entry. A prefix-less row therefore NEVER owns ANY model, so `probe` is always true and `serve` (src/run/serve.rs:69) prepends a list-models GET before encode. For openai-chatgpt that GET targets `https://chatgpt.com/backend-api/codex/models`, which 400s ("Field required: query.client_version" — codex has no standard /models listing), so every canonical-path generation now dies at exit 69.

## Repro (live, openai-chatgpt row in ~/.config/brazen/config.toml)

The EXACT invocation documented as working in memory bz-live-openai-chatgpt-testing.md:
```
printf {body} | bz --provider openai-chatgpt --input req.json --text
# req.json = {"model":"gpt-5.4","system":[...],"messages":[...],"stream":true,"store":false}
```
Observed: `models request failed with HTTP 400`, exit 69.
Expected: exit 0, the answer "ok".

Proof the row itself is healthy and only the probe is at fault: the same request via `--raw` (which skips the probe, serve.rs:69 `&& !raw`) with a correct openai_responses body round-trips end-to-end on codex (gpt-5.4 answered "ok", exit 0). So the regression is exactly the unconditional probe on a prefix-less row, NOT codex connectivity/auth.

## Scope

- openai-chatgpt (custom OAuth row): every canonical generation now fails 69.
- openai-responses (shipped defaults.toml row, no model_prefixes by design): same flaw — any `bz --provider openai-responses --model gpt-4o-mini "..."` now fires an extra/needless list GET. (Against real OpenAI /v1/models that GET may 200, so it would silently add a round-trip rather than fail, but it is still wrong: a fully-specified exact id should never need a probe.)

## Fix direction (for the implementer)

A fully-qualified exact model id should not require a probe regardless of row prefixes. Options: treat a non-empty explicit `--model`/request-model as owned (probe only when the model is absent OR genuinely partial), or gate the probe on the model NOT already being an exact member — but membership is only known after the list GET, so the cleaner invariant is: probe iff the routing model is absent. A prefix-less row reached via explicit `--provider` with an explicit model should skip the probe. Confirm against the model-discovery spec (the §5.1 owned-vs-probe query) and fix the doc if it mandated the current behavior.

Separately worth a note (NOT this ball): `bz list-models --provider openai-chatgpt` also 400s for the same codex /models reason — list-models is simply unusable on that backend; an honest provider limitation.
