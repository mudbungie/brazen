Owner ruling 2026-07-08: 'native-certs is great, the default should be off, of course (secure defaults).' Two parts, one lane (src/native/ + Cargo.toml):

1) Error-chain surfacing: native/transport.rs collapses every ureq failure — DNS, connect, TLS handshake, cert rejection, timeout, reset — into one opaque ErrorKind::Transport whose message is e.to_string() (~lines 78, 120-126). A user behind a TLS-inspecting corporate proxy cannot distinguish 'cert not trusted' from 'host down'. Fix: include the full source chain in the Transport error message (walk std::error::Error::source, join with ': '). ONE ErrorKind::Transport stays — no taxonomy change, no new exit code; this is message quality only.

2) native-certs cargo feature: Cargo.lock pins ureq 3.x with webpki-roots only (bundled Mozilla roots); no OS cert store. Add a cargo feature 'native-certs' (rustls-native-certs / ureq's equivalent feature — check ureq 3's feature surface first), DEFAULT OFF (secure defaults per owner ruling). Enterprises building from source flip one feature to trust the OS store (corporate roots). No runtime config surface — it is a build property. Document in README build/install section + architecture.md §10 portability + §12 (the current bundled-roots limitation is undocumented; write it down either way). Feature code lives entirely in src/native/ (the coverage-excluded impure zone) — verify tests/purity.rs and the coverage exclusion are unaffected; add a compile-check for the feature in CI/Makefile if the Makefile has a feature matrix (do not build a matrix if none exists — one 'cargo check --features native-certs' line suffices).
