The three dialects landed in bl-2f13 map a POST-200 mid-stream error to ErrorKind::Transport (exit 69) unconditionally:
- openai_responses: `response.error`/`response.failed` (src/protocol/openai_responses/decode.rs, fn stream_error)
- ollama_chat: a mid-stream `{"error":...}` line (src/protocol/ollama_chat/decode.rs, fn stream_error)
- google_genai: (no mid-stream error path implemented at all)

This DIVERGES from providers.md §3.7/§4.8 (and the §8/CR-10 contract), which specify a mid-stream error is exited by its DECODED kind: a 5xx-class error.type -> Provider{>=500}/70, rate-limit -> Provider{429}/69, otherwise Transport/69. AnthropicMessages already does this (src/protocol/anthropic/decode.rs, fn error_kind). The new dialects are inconsistent with that established pattern.

Decision needed: either (a) implement the by-kind mapping for these dialects to match the spec + Anthropic, or (b) if Transport-default is genuinely acceptable, fix providers.md §3.7/§4.8/§5.9 to say so. Per AGENTS: don't leave a silent code-vs-spec deviation. Low urgency (post-200 mid-stream errors are rare), but it is a real divergence, not a documented deferral.
