Arch-review finding (2026-07-07). Structured output is table-stakes for harness callers (typed tool results, extraction, judge verdicts), and it meets the EXACT test that lifted reasoning (architecture.md §3.1: 'every dialect names the same idea under an irreconcilable spelling; extra can carry only one spelling'): OpenAI chat/responses response_format {type:json_schema,json_schema:{name,schema,strict}} (+ text.format on responses), Google generationConfig.responseMimeType:application/json + responseSchema, Ollama format:json|schema-object, Anthropic output_format/structured-output (verify current wire; historically tool-forcing was the idiom). Today every one of these rides extra verbatim (openai-chat-mapping.md:71-73 lists response_format as 'no canonical home'; providers.md:230 Gemini responseSchema via extra; providers.md:342 Ollama format via extra) — the caller must know each dialect's spelling, defeating the abstraction.

DESIGN-FIRST, spec in architecture.md §3.1 before dialect specs. Candidate shape: req.output: Option<OutputFormat> with OutputFormat::{JsonSchema{name,schema,strict?}, Json} (lean minimal — two variants; plain text = None, the empty-set path). Each encode owns the projection; typed knob wins over a same-named extra key through the existing one-fold rule. Dialects without the concept: unsupported_body_keys opt-out precedent (config §4.1.1) or ParseInput — argue from the reasoning-knob precedent (providers.md §6). Decide interplay with Tool::Custom.strict (OpenAI strict function calling is a per-tool sibling of response_format — same spec, possibly Tool::Custom gaining Option<strict>).

Follow the bl-24e5/bl-1421 pattern: this is harness-support, motivated by lernie. Do not gold-plate: no streaming-validation, no client-side schema checking — brazen carries the intent, the provider enforces it.
