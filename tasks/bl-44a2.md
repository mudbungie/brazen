Whether corporate-proxy users work AT ALL is currently unknown.

THEN decide the stance per the empty-set rule and spec it in architecture.md §10 (portability/deployment) — one of: (a) ureq honors the env vars with current features → document exactly that, plus which schemes, and add a test if cheap (the sim-conformance FakeProvider loopback precedent could host a fake CONNECT/absolute-URI proxy — weigh cost honestly; a native-zone test may suffice); (b) ureq needs a feature brazen doesn't enable → decide enable-by-default vs a cargo feature (mirror the native-certs precedent bl-770f: default off unless the behavior is pure-additive and expected — proxy env honoring IS the ecosystem default expectation, lean enable); (c) ureq does nothing with env → decide document-as-unsupported vs implement env reading at Agent build (severability: env-driven, no new flags — NO --proxy flag either way; the env vars are the ecosystem's existing explicit signal).
