Found while delivering bl-0272 (folded out of that auth task to keep it auth-focused). The reasoning circuit works LIVE for openai-chatgpt (verified 2026-06-17) but NO live harness pins it — `bz/tests/live_fuzz_openai.rs` (bl-b72f) drives request/error shapes, not reasoning.

## What to add
A token-costing acceptance case in `live_fuzz_openai.rs` (`accept_cases()`, the `BRAZEN_LIVE_FUZZ_SPEND=1` set) that REQUESTS reasoning and ASSERTS the decoded canonical grammar. The request carries `reasoning:{effort:..., summary:auto}` (rides the request `extra` flatten, like `store`); without the `reasoning` field codex emits NO reasoning at all.

## The nuance to VERIFY live before asserting
For this backend (ChatGPT-SSO / codex, gpt-5.4) only the SUMMARY channel fires: `response.reasoning_summary_text.delta`. The RAW `reasoning_text.delta` channel (bl-7e50) and the bl-d884 reasoning/text "coexistence" case were NOT observed live on 2026-06-17.

OPEN QUESTION the implementer must settle by driving it live (do not assume): what canonical events does brazen`s openai_responses decoder map `response.reasoning_summary_text.delta` to? Candidates: a `content_start{thinking}` + `thinking_delta` (the task note bl-0272 guessed this), OR a distinct summary kind. Capture the ACTUAL `--json` event stream and pin the grammar that really decodes — assert the observed shape, not the guess. If the summary channel does NOT decode to thinking events, that is itself a finding (decoder gap → its own fix-ball).

## Shape / gating
Reuse the existing `accept_cases()` + `grammar_ok()`/`check_accept` machinery and the `is_tool`-style per-case grammar selector (extend it to a reasoning variant). GENERATES, so SPEND-gated like the other acceptance cases. Keep `live_fuzz_openai.rs` <300 lines (precommit hard cap) — the shared helpers live in `bz/tests/live_support/openai.rs` (bl-f8f7) + `exec.rs`/`grammar.rs`.
