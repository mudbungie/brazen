Motivation: lernie v0.4 (ARCHITECTURE §2.10/§4.4) moves transient-error retry into the harness, exactly as brazen intends (non-goal §2: brazen exposes retryable() but never acts on it). A caller-owned retry loop pacing 429/529 wants the provider's Retry-After — an HTTP response HEADER, i.e., transport knowledge brazen holds and the caller cannot recover: provider_detail carries the parsed error BODY verbatim, and the header is not in the body. AGENTS.md carry-the-fact: 'If a component already knows a fact (the transport knows the HTTP status), thread it through to whoever needs it rather than re-deriving it downstream from a stand-in' — the same argument that put Frame.status on the whole-body frame (CR-10).

Verify first (never assume docs correct): confirm the transport/Frame path actually drops response headers today; if some path already surfaces them, spec that instead of adding a field.

Proposed shape, spec first: CanonicalError.retry_after_seconds: Option<u32>, populated from the Retry-After header on non-2xx handshakes — both integer-seconds and HTTP-date forms (date computed against the injected Clock seam, never wall-clock reads); None where absent (empty-set rule — never fabricated). Additive under the error schema's grows-only tolerance (§3.2 v=1 contract: the error event has no version gate and must grow additively; a new optional field with serde default reads old lines as None). The --json error line gains the optional key; v=1 consumers already must ignore unknown fields. Plumbing: header capture on Frame (sse-decoder.md §9) or alongside frame.status, mapped once where ErrorKind::from_http_status runs. Spec surfaces: architecture.md §3.3/§8, sse-decoder.md §9, providers.md. Distinct from provider_detail (body, verbatim diagnostics) — this is the transport-level fact, carried.
