SHIPPED. Rung 2 (last-used) built; the ladder is now complete.

THE LADDER (model-discovery.md 4.1, written this ball):
  1. configured model -- INTENT (--model / BRAZEN_MODEL / config `model`). A
     non-empty seed never reaches rungs 2-3 at all. ABSOLUTE.
  2. last_used -- OBSERVATION. Consulted ONLY on an empty seed, and only if the
     list still carries the id.
  3. the provider's own suggestion: its `default`-flagged entry, else models[0].

Rung 2 was placed ABOVE the `default` flag as well as above the head. Rung 3 is
a list position nobody chose; a model you actually reached for is the nearest
thing to intent that exists absent config. This matches the user's verbatim
"the cached answer of last-used wins, and WORST-CASE ... whatever the provider
lists at the top of its list".

RULING 1 -- STORAGE: the recommended location was adopted unchanged. `last_used`
is a key in the existing per-provider cache file
$XDG_CACHE_HOME/brazen/models/<provider>.json. `ModelCache` now gets/puts one
`CachedModels { models, last_used }` document instead of a bare Vec<Model> --
one file, one document, one serde, no second read path. The key is ADDITIVE
(serde(default) + skip_serializing_if), so a pre-pointer cache reads clean to
None and a pointer-less document serializes byte-identically to the old
{"models":[...]} shape: no cache-version field, no break, the same grows-only
v=1 discipline the metadata keys ride.

REJECTED ALTERNATIVE (recorded in the spec): reusing `Model.default` as the
pointer. It needs no new key and select_model already reads it -- but `default`
is a CARRIED PROVIDER FACT (3), so writing a local observation into it makes
`--list-models --json` report a default the provider never declared, and the
wholesale relist silently wipes it. Two facts, two homes.

RULING 2 -- LEARN-ON-SUCCESS OVERLAP: they LAYER, they do not subsume. The
preferred subsumption was examined and rejected as DISHONEST, because they are
two facts, not two copies of one:
  - the tail append records MEMBERSHIP ("this row can serve this id") -- read by
    partial matching (4) and by routing's ownership tier (config.md 7 step 3b);
  - last_used records RECENCY ("this is the one you reached for") -- read only
    by the empty-seed ladder.
Folding the append into the pointer would DELETE the membership fact: a learned
id would stop being partial-matchable and stop being a routing fact. That is a
loss dressed as a simplification.

What the pointer DOES subsume is the append's old DEFAULTING duty. 5.4 used to
lean on "on a cold cache the learned id is the lone entry, therefore the
default" -- a coincidence that held only in the degenerate single-entry case.
Defaulting is now rung 2's job outright and the append is purely membership.

The AGENTS.md drift hazard is answered STRUCTURALLY rather than by deletion:
ONE write site, ONE invariant -- after any successful write, `last_used` names
an id present in `models`. The pointer always points into the list it was
written with, so the two can never diverge. Consequent write rules:
  - EVERY 2xx moves the pointer (using a listed model is still using it);
  - only a VERBATIM 2xx appends (unchanged, dedup-free);
  - an UNCHANGED document is never written, so the steady state (the same model
    call after call) costs no file write at all. Concurrent bz processes race
    only on an atomic rename, and last-writer-wins is the right semantics for
    "last used".

CONSTRAINTS HELD:
  - No reorder. The pointer sits BESIDE the list; 5.4 "append, never reorder" is
    untouched, so there are never two representations of "which model is first".
  - Rung 2 never overrides rung 1 -- verified against the real binary.
  - Forgiving: a DANGLING pointer (id no longer in the list) or an empty one
    falls straight through to rung 3, never an error, and an empty list is still
    the lone Config (78) even with a pointer set -- the pointer must resolve INTO
    the list and can never conjure a model. A WRONG-TYPED last_used is file
    corruption like any other (whole document -> None, self-healing on the next
    --list-models), which is exactly cache.rs's existing rule, unchanged.
  - --list-models REPLACES the list and CARRIES the pointer forward: re-listing
    changes which ids exist, never which one you last used.
  - PROVIDER selection untouched. No last-used anywhere in routing; bl-834f's
    two-pass fall-through was not modified.
