SUPERSEDES this ball's original framing. The earlier body posed
"top-of-config VS last-used" as a conflict needing a ruling. THAT WAS A
MISREADING. The user restated the model on 2026-07-19 and there is no
conflict -- they are rungs of one ladder.

THE USER'S MODEL (verbatim, 2026-07-19):

  "the best match to the user's request is served, with priority given to
  actual config (because it reflects intent). So, if I say, `bz --model opus
  yo`, normally, we would just read the config, but the user specified a
  model, so every config combination, cascading down through the
  possibilities, which does not satisfy the request, is thrown out, and the
  next tested against the criteria, and tried. if the user says, --provider
  anthropic, but has never actually configured any model to be their default,
  then the cached answer of last-used wins, and worst-case, the answer of
  whatever the provider lists at the top of its list wins. If they say --model
  5.5, and their first provider is anthropic, it fails, and their second
  provider, openai, would be the greedy match."

RESTATED: constraints FILTER, priority RESOLVES. --model/--provider are not
selectors, they are filters; every (row, model) possibility failing them is
thrown out, and resolution cascades down config order taking the first
survivor. Config outranks all else BECAUSE IT ENCODES INTENT.

THE MODEL-SELECTION LADDER (this ball's whole scope):
    1. configured model   (intent -- config file / --model / BRAZEN_MODEL)
    2. LAST-USED          <-- THE MISSING RUNG. This ball builds only this.
    3. provider's first-listed model  (today's fallback)

PROVIDER SELECTION IS OUT OF SCOPE AND NEEDS NO LAST-USED. Every provider
example the user gave resolves by config order + filtering; last-used never
has to break a provider tie. Their only last-used example is explicitly
model-scoped ("--provider anthropic, but has never configured any model").
This drops the cross-invocation non-determinism worry for provider entirely --
do NOT reintroduce it.

THE GAP IS REAL (verify before building): src/canonical/model.rs select_model
with seed == "" goes straight to first `default`-flagged else models[0]. No
dialect sets `default`, so in practice models[0]. Learn-on-success
(src/run/generate.rs) APPENDS a used id to the cache TAIL -- and
model-discovery.md 5.4 rules "Append, never reorder... learning only fills a
gap" -- so a model you actually used becomes the empty-seed default ONLY in
the degenerate cold-cache single-entry case. Rung 2 genuinely does not exist.

WHERE LAST-USED LIVES (recommended, not yet user-confirmed): a `last_used`
field inside the EXISTING per-provider cache file
$XDG_CACHE_HOME/brazen/models/<provider>.json (src/native/cache.rs). Rationale:
already per-provider; already the home for observed-not-configured facts; no
new state file; no new global mutable thing for the user's parallel agents to
race on. Alternatives the user may prefer: per-cwd, or per-config-file. DO NOT
invent a third option without saying so loudly.

CONSTRAINTS ON THE BUILD:
  - Do not reorder the cached list. model-discovery.md 5.4's "append, never
    reorder" stands; last_used is a POINTER beside the list, not a
    permutation of it. Two representations of one fact would drift.
  - Rung 2 must not override rung 1. An explicit --model or a configured
    model always wins; last_used is consulted ONLY on an empty seed.
  - Interaction with learn-on-success: they now overlap. Decide whether
    last_used SUBSUMES the tail-append (preferred if honest -- one memory of
    "what you used", not two) or layers on it. AGENTS.md: "two representations
    of one fact will drift."
  - Forgiving read: a missing/corrupt last_used must degrade to rung 3, never
    error (cache.rs:56-63 already returns None on any corruption).
