USER ASK (2026-07-19, verbatim): "while unspecified, the provider should be
top-in-config-selected. while unspecified, the model should be
top-in-config-selected. while any of these are implicit (autodetected), the
ruling should be last-used."

THE FIRST TWO CLAUSES ARE ALREADY RULED AND IMPLEMENTED -- see bl-942d
("zero-config default IS .first()", "empty model seed takes select_model's
first cached model"). This ball is ONLY the third clause.

THE TENSION, STATED PLAINLY: "unspecified -> top-in-config" and "implicit ->
last-used" are two different answers to the SAME question (what happens when
the user names nothing). They cannot both be the rule.

INTERPRETATION TAKEN (flagged as an interpretation -- confirm before building):
top-of-config is the COLD-START default, used when there is no history. Once a
provider/model has been resolved implicitly at least once, that resolution is
remembered and outranks top-of-config, until an explicit --provider/--model
overrides it (which then becomes the new remembered value). Explicit always
wins over remembered; remembered wins over top-of-config.

WHY THIS NEEDS A RULING, NOT JUST AN IMPLEMENTATION:

  1. It introduces PERSISTED SELECTION STATE, which the repo currently has
     none of. Verified by grep: no last_used|last_provider|state.json|recent
     anywhere. The only persisted state today is the model cache
     ($XDG_CACHE_HOME/brazen/models/<provider>.json) and the CredStore -- both
     are CONTENT (what exists), never PREFERENCE (what you chose last).
     This is a new category of state.

  2. It cuts against AGENTS.md: "Single source of truth. Every fact has one
     authoritative home... don't store what you can compute -- make it a
     query, not a field". Last-used is by definition not computable; it is a
     stored fact with no derivation. If it is built, the ball must say why
     this fact earns storage where others did not.

  3. It makes bz NON-DETERMINISTIC ACROSS INVOCATIONS. Today `bz "q"` with a
     given config always resolves the same way. After this, the same command
     in the same config resolves differently depending on invocation history
     -- including history from another terminal, another project, or a
     background agent. In a repo whose users run many parallel agents (see
     the bl worktree workflow), a shared mutable last-used file is a race and
     a surprise vector. Scope question the ball MUST answer: global, or
     per-cwd, or per-config-file?

  4. Interaction with learn-on-success (src/run/generate.rs:113-122). That
     already appends a Verbatim model id to the cache tail on 2xx, which on a
     COLD cache makes the learned id the empty-seed default -- a de-facto
     last-used in the single-entry case. Does last-used SUPERSEDE that
     mechanism, or layer on top of it? Two overlapping memories of "what you
     used" is exactly the drift AGENTS.md warns about. Strongly consider
     whether last-used should REPLACE learn-on-success rather than join it.

  5. Discoverability. If bz silently prefers a remembered provider, the user
     cannot see why. bl-942d already accepted this cost once ("the pick is not
     surfaced... no --why-this-row added"). Doing it twice compounds it. Does
     --dump-config show the remembered value?

DELIVERABLE: a written ruling in specs/ (architecture.md 4.3 + config.md 7 are
the homes -- the same sections bl-942d and the fall-through ball touch) BEFORE
any code. If the ruling is "don't build it, learn-on-success plus
top-of-config is enough", that is a valid and cheap outcome -- say so and
close.

SERIALIZED after the fall-through ball: both edit route() in
src/config/resolve/mod.rs and the same spec sections. Parallel lanes here
would collide.
