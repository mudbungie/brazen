## Symptom
`bz --provider ollama --model M --stream` and the same command WITHOUT `--stream` send the byte-identical wire request (verified live: same event output, same exit). The only way to get `stream:true` on the wire is to put `"stream":true` in the canonical request JSON on stdin. The `--stream` flag (and `BRAZEN_STREAM` env, and config `stream=`) do nothing to the request.

## Root cause
`--stream` sets `cfg.stream = Some(true)` (src/cli.rs:79), carried into `ResolvedConfig.stream` (src/config/resolve.rs:40, resolved.rs:29). But NOTHING reads it: `fill_absent` (src/config/resolved.rs:66) sets model/max_tokens/temperature/top_p/system — NOT stream. Every encoder serializes `req.stream` (e.g. openai_responses/encode.rs:42 `body.insert("stream", json!(req.stream))`), and `req.stream` defaults to `false` (canonical/request.rs:43 `#[serde(default)] pub stream: bool`). So config.stream is dead.

## Fix sketch
`fill_absent` should fold cfg.stream into req.stream (the request body wins when it explicitly sets stream; else the resolved flag applies). One line: `req.stream = req.stream || cfg.stream.unwrap_or(false);` — or, if request-JSON stream should always win when present, model 'absent' explicitly (Option<bool> on the request) so the flag fills it like every other field. Either way: single source of truth = the resolved config, filled into the request exactly like model/max_tokens.

## Impact / note
bz/tests/ollama_smoke.rs passes `--stream` and claims it 'exercises the real streaming frame path' — it does not; ollama receives stream:false and returns a single buffered response. The live conformance suite (bl-04dc) sets `"stream":true` directly in the request JSON to actually stream.

Found by bl-04dc (live conformance suite).
