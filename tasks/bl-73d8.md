Validated live 2026-06-16 against https://chatgpt.com/backend-api/codex/responses (model gpt-5.4, ChatGPT Business account).

brazen `openai_responses` encode renames canonical `max_tokens` -> `max_output_tokens` (src/protocol/openai_responses/encode.rs:34, per Responses spec §3.2). The Codex backend REJECTS that field: a body carrying `max_output_tokens` returns `400 {"detail":"Unsupported parameter: max_output_tokens"}`. The same body with the field omitted streams a 200 response normally.

Consequence: `bz --provider openai-chatgpt --max-tokens N ...` (or any input JSON with `max_tokens`) ALWAYS 400s — and because of the non-2xx body swallow (bl-5fe6) the user sees only an empty error.

brazen encodes CORRECTLY per the standard OpenAI Responses spec; the Codex backend is the non-standard party (it also forbids missing `instructions`/`store:false`/`stream:true`). So the question is whether the openai-chatgpt ROW should suppress `max_output_tokens`. Options, lightest first:
- Document the limitation (operator omits --max-tokens for this row). Zero code.
- A per-row data flag to drop unsupported fields (only if a second field ever joins it — do not add speculatively, architecture.md §4.6 / AGENTS.md severability).

Recorded in specs/auth.md §10.7. Relates to bl-5fe6 (the swallow that hides the detail). Discovered during the §10.7 live validation (bl-1444).
