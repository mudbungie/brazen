ureq is in [dependencies] (Cargo.toml:29), not bin-scoped — Cargo has no per-binary-target dependency gate (unlike [target."cfg(unix)".dependencies] for libc, which is platform-scoped, not target-scoped). So the lib *can* `use ureq`; nothing but a comment + discipline stops a future lib module from linking the network client, silently breaking the "lib is pure / network-free / 100%-covered behind injection" invariant (arch §9.5, §10; the #![forbid(unsafe_code)] / pure-core posture in lib.rs).

Today the only ureq user is HttpTransport in src/bin/main.rs (the coverage-excluded bin, landed in bl-838c). The invariant holds by convention, not by build.

Fix options (pick when worth the ceremony):
- Split bz into its own workspace member crate depending on brazen (lib) + ureq; the lib crate then literally cannot link ureq. Cleanest.
- A clippy/CI guard that greps lib sources for ureq / network crates.

Low urgency: no current violation. Filed so the gap is tracked rather than silently load-bearing on discipline.
