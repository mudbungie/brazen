## Problem
`bz yo` (a bare prompt, no `--provider`, no `--model`, no `BRAZEN_MODEL`) exits 78
`no provider resolved`. With no specification at all it should Just Work: route to
the FIRST provider in the config and let the model default to the FIRST cached model.

## Current behavior
`PartialConfig::route` (src/config/resolve.rs) returns `ConfigError::NoProvider` when
no `--provider` is named AND there is no routing model (`routing_model = None`):
    let model = routing_model.ok_or(ConfigError::NoProvider)?;
The model half is ALREADY solved: `select_model` (src/canonical/model.rs) maps an
empty seed `""` to `models[0]` (the first cached model). The gap is ONLY provider
selection when nothing is specified.

## Fix
In `route`, when no provider is named AND `routing_model` is None, default to the
FIRST provider row instead of erroring. `providers` is a `BTreeMap<String,_>`, so
"first" = first by provider NAME (lexicographic) — deterministic, config-driven
(rename to control it). For the default config that is `anthropic` (also file-first).
An EMPTY provider table still → NoProvider (truly nothing to route to). A model GIVEN
but owned by zero rows STILL → NoProvider (arch §4.3 `bz -m opus "q"` invariant intact)
— only the no-model-AND-no-provider case changes.

## Login exception (credential WRITE must name its target)
`route` is shared by 4 callers of `into_resolved`: generate (data plane), `--raw`,
`--list-models`, and `--login`. The first three (read/send) SHOULD default. `--login`
WRITES a credential under a provider key — auto-picking the target is unsafe and
useless (the default provider rarely has oauth). Keep login strict: guard
`merged.provider.is_none() -> NoProvider` in `resolve_oauth` (src/auth/login.rs) so
`bz --login` with no provider keeps its clean 'no provider resolved' error.
