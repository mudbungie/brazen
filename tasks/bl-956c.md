Arch-review finding (2026-07-07). Content (canonical request, src/canonical/request.rs) has exactly one non-text media variant: Image{ImageSource}. PDFs/documents — a first-class content type on every major wire (Anthropic {type:document, source:{base64|url|text|file_id...}} incl. citations config; OpenAI chat/responses file parts {type:file/input_file, file_data|file_id}; Gemini inlineData/fileData with application/pdf mimeType) — cannot be expressed at all: content blocks have NO extra valve (the flatten valve is top-level-request-only), so this is hard-absent, and nothing in specs records a deferral (grep pdf/document/audio comes back empty on the input surface). A harness feeding user files (the -f flag's own forward note, architecture.md §5.5: 'a follow-up could detect an image file and emit Content::Image') will hit this immediately.

DESIGN-FIRST in architecture.md §3.1. Candidate: Content::Document{source: DocumentSource{Base64{media_type,data}|Url{url}}} mirroring ImageSource exactly (one new variant, one new source enum — additive to the wire-request parse; the same reject-at-encode rule as Image for dialects whose wire lacks it). Explicitly decide and record the AUDIO question in the same spec pass (probably a documented deferral — the empty-set rule needs the rationale written, not silence). Consider whether -f should detect PDFs (media-type by extension/magic) once the variant exists — separate ball if yes, note it.

Do NOT add a per-block extra valve without separate design scrutiny — that is a bigger door (it would also be where per-message name/metadata pressure goes; note the tension for the spec).
