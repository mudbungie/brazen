Owner ruling 2026-07-08: 'sure, I can't think of a strong reason not to do this.' architecture.md §5.10.2/§13.14 already sanctioned the split as forward-compatible: bare --raw means both directions today and forever, so --raw=in/--raw=out is a pure backward-compatible extension. The compelling case is normalized-in/raw-out: bz's request ergonomics (positional prompt/-f, config fold, model cache, auth) with the EXACT provider wire bytes out — today impossible (raw-out forces hand-writing the whole provider-native request), and the only encode-observability window since there is deliberately no --debug flag (§2).

SPEC-FIRST (architecture.md §5.4/§5.10.2/§13.14 + the flag table §5.10.3): define the value grammar (--raw bare = both, unchanged forever; --raw=in = verbatim request in, canonical events out; --raw=out = canonical request in via constructor/stdin, verbatim response bytes out; decide whether --raw=both is an accepted explicit spelling). Rules to pin per direction: raw-in keeps stdin-verbatim semantics (no constructor, -f refused as today — re-derive the §5.5 composition rules per direction and write them); raw-out keeps the §5.4 output rules (provider terminator stands, no injected end, RawSink per chunk) while normalized-out keeps the one End; the raw 4xx/5xx-never-exits-0 rule holds in every combination; --json/--text interaction (an output MODE question: --raw=out IS the output mode — spec the mutual exclusion consistently with the existing last-wins OutMode fold). Implementation: §13.14 records the halves already toggle independently in serve/drive — make that real: the encoded-request path with RawSink output, and the verbatim-request path with the framed decode. Tests: all four combos end-to-end via MockTransport + run_modes/run_inputs suites; smoke.sh --raw probes still pass unchanged.
