read_request (src/pipeline/input.rs:34) drains stdin with reader.read_to_end on the positional-prompt path to enforce the prompt-XOR-stdin rule. When stdin is an interactive terminal (or any open pipe with no EOF), read_to_end blocks forever — so `bz "hi"` typed at a shell, or driven by a harness that leaves stdin open, hangs instead of running.

The drain is deliberate (input.rs:31-33 doc: "the prompt path still drains reader to prove it is empty"), but it assumes stdin reaches EOF, which an interactive tty never does.

Repro: `bz "hi" --provider anthropic ...` with stdin a tty -> hangs. Workaround today: append `</dev/null`. Surfaced while smoke-testing HttpTransport (bl-838c): all three live tests needed </dev/null.

Design (needs a small spec note): when stdin is interactive, treat it as absent rather than draining it. Detection is isatty(0) — an impurity, so it cannot live in the pure lib (input.rs). Options: compute "stdin is a tty" in the bin (libc, like the SIGPIPE restore) and pass it into read_request (a bool, or only hand read_request a stdin reader when non-tty), keeping the XOR check (exit 64) for the genuine piped-stdin-plus-prompt case. Keep the lib pure — the tty probe is a bin/seam concern.
