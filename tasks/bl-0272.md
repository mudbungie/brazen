bl-b72f (OpenAI ChatGPT-SSO fuzz) scoped "Auth/refresh fuzzing: near-expiry token -> silent refresh; revoked -> exit 77" but delivered only the request/stream/error-conformance half. The auth circuit was left untested because it needs deliberate token manipulation (the live cred on this box is fresh: exp ~1782535682 / 2026-06-27).

## Circuits to plumb (brazen, not OpenAI)
1. SILENT REFRESH (auth §6): with an EXPIRED/near-expiry access token but a valid refresh_token, a data-plane `bz --provider openai-chatgpt ...` must transparently mint a new access token (token_url POST), persist it back to the XdgCredStore, and complete the request 200. Today only offline (tests/oauth_refresh.rs, MockTransport) and the interactive oauth_smoke.rs exercise refresh — never the codex row live.
2. REVOKED -> 77: with a revoked/invalid refresh (or access) token, the failure must map to ErrorKind::Auth -> ExitClass::NoPerm -> exit 77 (codex returns 401/403; from_http_status maps 401|403 -> Auth). NOTE: my bl-b72f error matrix is ALL 400 -> 69; the 401/403 -> 77 mapping is UNTESTED live for this provider.

## Test approach (no real-cred risk)
Work on a COPY of the cred JSON in a temp XDG_DATA_HOME: (a) rewrite `exp` to the past to force refresh; (b) corrupt the refresh_token to force the revoked path. Drive `bz` with that temp store. Keep it #[ignore]d + BRAZEN_LIVE-gated like bz/tests/live_fuzz_openai.rs; reuse the same harness leaves. Relates to bl-5b5a (oauth smoke), tests/oauth_refresh.rs.
